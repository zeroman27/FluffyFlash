# UI Design Brain (Wist)

Локальная копия навыка [carmahhawwari/ui-design-brain](https://github.com/carmahhawwari/ui-design-brain): **смысловые** паттерны интерфейса (когда что применять, иерархия, формы, модальные сценарии, таблицы, пустые состояния). Полный справочник: **[components.md](components.md)**.

## Как не дублировать другие skills в этом репозитории

| Навык | Роль | Не дублировать |
|--------|------|----------------|
| **swiftui-pro** | Ревью и реализация **Swift/SwiftUI**, HIG, API, a11y, перфоманс | Паттерны «что за компонент» — это Design Brain / UX Max, не дублировать правила из `references/design.md` в новых экранах без ссылки на задачу |
| **ui-ux-pro-max** | **Поиск** дизайн-системы (`search.py --design-system`), стили, палитры, стеки; есть `--stack swiftui` | Не перечитывать те же «общие профессиональные правила» из Design Brain как второй источник истины — сначала UX Max для системы, Brain для **компонентного** разбиения экрана |
| **ui-design-brain** (этот) | **Семантика** экрана: какие блоки, порядок, состояния, анти-паттерны по компонентам | Не генерировать React/Tailwind по умолчанию — проект **macOS SwiftUI** |

**Порядок работы для Wist:**  
1) По запросу «как устроить экран» — открыть релевантные разделы **components.md**.  
2) По запросу «палитра / стиль / landing-структура» — **ui-ux-pro-max** (`search.py`, при необходимости `--stack swiftui`).  
3) По запросу «как написать View / проверить код» — **swiftui-pro**.

## Адаптация под macOS (не веб)

Оригинальный skill ориентирован на HTML/React. Для **Wist** переносите идеи так:

| Идея из справочника | В SwiftUI / macOS |
|---------------------|-------------------|
| Modal / диалог фокуса | `.sheet`, `confirmationDialog`, `alert`, отдельное окно при необходимости |
| Drawer | `NavigationSplitView` колонка, `inspector`, широкий `sheet` |
| Dropdown / меню | `Menu`, `Picker`, контекстное меню |
| Table | `Table` / `List` с колонками, виртуализация для длинных списков |
| Toast | нативно нет 1:1 — краткий текст в UI, `ProgressView`, или ненавязчивый баннер в контенте (без дублирования системного Alert) |
| Hover / pointer | правила про `cursor-pointer` из **ui-ux-pro-max** к **вебу**; на Mac — hover-модификаторы и курсоры SwiftUI при необходимости |

Минимальная площадь нажатия на Mac — ориентир **не меньше** системных рекомендаций Apple (часто ≥ 44 pt эквивалент для кликабельных зон; см. swiftui-pro / HIG).

## Workflow (сокращённо)

1. **Разобрать запрос** — какие компоненты по смыслу (навигация, форма, таблица, пустое состояние, прогресс).  
2. **Открыть** [components.md](components.md) по имени компонента или по разделу «feedback / input / …».  
3. **Применить** best practices к структуре экрана; **реализовать** в SwiftUI согласно **swiftui-pro**.  
4. **Стиль и токены** — при необходимости согласовать с **ui-ux-pro-max** и существующими токенами приложения (`WistTheme` и т.д.).

## Анти-паттерны (сохраняют силу)

Дублирование модальных уровней без причины, равновесные кнопки без primary, плейсхолдер вместо подписи поля, спиннер вместо скелета для предсказуемой вёрстки, мелкий текст как основной контент — см. раздел anti-patterns в оригинальном SKILL и детали в **components.md**.
